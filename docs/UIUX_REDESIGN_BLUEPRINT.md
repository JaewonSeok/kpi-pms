# 평가시스템 전체 UI/UX 개편 설계서

Last updated: 2026-06-17

## 1. 개편 목표

이번 개편의 목표는 KPI PMS를 "기능이 많은 관리 도구"에서 "사용자가 지금 무엇을 해야 하는지 바로 이해하는 평가 운영 시스템"으로 바꾸는 것이다. 7/1 MBO/KPI 오픈과 이후 평가 운영을 기준으로 다음 흐름이 자연스럽게 이어져야 한다.

1. 본부 KPI Alignment
2. 개인 KPI/MBO 작성
3. 월간 실적, 증빙, 코멘트 축적
4. 리더 검토와 확정
5. HR readiness와 blocker 관리
6. 수행결과 작성
7. 팀장 1차평가
8. HR 본부검수
9. 최종평가와 대표이사 보고
10. 조직 30% + 개인 70% 점수 preview
11. 등급 preview
12. 공식 저장 차단과 안전 잠금

핵심은 더 예쁜 화면이 아니라, 역할별로 해야 할 일을 줄이고 순서를 분명히 보여주는 것이다.

## 2. 현재 UI 문제 진단

| 영역 | 현재 문제 | 사용자 영향 |
| --- | --- | --- |
| 전체 구조 | 기능 단위 메뉴가 많고 평가 흐름 순서가 약함 | 사용자가 다음 행동을 찾기 어렵다 |
| 상단 영역 | 큰 안내 박스와 넓은 요약 영역이 반복됨 | 실제 입력/검토 영역이 아래로 밀린다 |
| 우측 패널 | 긴 설명, 세부 옵션, 보조 정보가 한 번에 노출됨 | 화면 피로도가 높고 핵심 CTA가 묻힌다 |
| KPI 입력 | 모든 필드가 같은 중요도로 보임 | 필수 입력과 선택 입력이 구분되지 않는다 |
| 월간 실적 | 매월 상세 입력을 요구하는 느낌이 강함 | 실사용자가 입력 피로를 느낄 수 있다 |
| 평가 화면 | 구성원/팀장/본부장/HR의 해야 할 일이 섞여 보임 | 권한별 책임이 흐려진다 |
| 관리자 화면 | 운영 링크와 고급 전환 기능이 함께 보임 | 실수 위험이 있고 데모 메시지가 분산된다 |
| AI 기능 | 작성 보조, 코칭, 리뷰 초안, AI 활용 제출이 섞여 보일 수 있음 | 공식 평가 AI처럼 오해될 수 있다 |

## 3. 전체 UX 원칙

| 원칙 | 설명 | 적용 방식 |
| --- | --- | --- |
| 지금 할 일 우선 | 사용자가 오늘 해야 할 일을 가장 먼저 보여준다 | 상단에 `다음 행동`, `마감`, `상태`를 배치 |
| 입력 부담 최소화 | 한 번에 모든 정보를 쓰게 하지 않는다 | 핵심 입력 -> 선택 입력 -> 증빙/메모 순서 |
| 흐름 기반 IA | 메뉴와 화면을 실제 평가 흐름 순서로 배열한다 | KPI 관리 -> 평가 관리 -> 관리자 설정 |
| 역할별 화면 | 구성원, 팀장, 실장, 본부장, HR의 일을 분리한다 | 같은 route라도 role-specific panel 사용 |
| 안전한 preview | 점수/등급은 preview와 official write를 명확히 분리한다 | `preview`, `안전 잠금`, `공식 저장 안 함` 칩 |
| Progressive disclosure | 고급 기능은 접고, 필요한 순간에만 연다 | accordion, detail drawer, collapsed section |
| 시각적 요약 | 표만 쓰지 않고 상태칩, 단계, 미니 차트를 쓴다 | Recharts, lucide icons, status chips |
| AI 역할 분리 | AI는 작성 보조, 코칭, 요약, 제출 증빙으로 분리한다 | 구성원 AI와 직책자 AI를 다른 영역에 배치 |

## 4. 디자인 톤앤매너

### 톤

- 업무 시스템답게 조용하고 명확해야 한다.
- 평가와 보상처럼 민감한 영역은 과도한 색상이나 장식보다 신뢰감을 우선한다.
- CEO/HR demo에서는 "현재 준비 상태와 안전 경계가 보인다"는 메시지가 중요하다.

### 화면 구성

- 큰 hero/card는 줄이고, 상단은 `상태`, `다음 행동`, `진행률`, `마감` 중심으로 구성한다.
- 반복 card는 8px 이하 radius를 기본으로 한다.
- 카드 안에 카드를 중첩하지 않는다.
- dashboard는 full-width band + constrained inner layout으로 구성한다.
- dense table은 요약 card, filter chip, sticky action bar와 함께 쓴다.

### 시각 요소

- 외부 이미지 의존성은 추가하지 않는다.
- `lucide-react` icon을 버튼/상태/빈 상태에 사용한다.
- 작은 흐름도, 진행 단계, 미니 bar chart에는 CSS와 Recharts를 우선한다.
- AI 기능은 반짝이는 장식보다 `보조`, `코칭`, `요약`, `preview` 라벨로 구분한다.

### 메뉴 운영 원칙

- `보상`은 7/1 MBO 오픈과 CEO demo nav에서 숨긴 상태를 유지한다.
- 공식 scoring/grade/department score intake 등 고급 전환 기능은 관리자 설정 안에서도 `고급` 또는 `공식 전환` 영역으로 접는다.
- 구성원 nav는 KPI 관리와 평가 참여만 보이게 한다.
- HR nav는 운영 허브를 첫 진입점으로 삼고, 세부 설정은 2차 링크로 둔다.

## 6. 역할별 홈/대시보드 개념

| 역할 | 첫 화면 메시지 | 우선 노출 |
| --- | --- | --- |
| USER | 내가 작성/제출해야 할 KPI와 월간 실적 | 개인 KPI 상태, 남은 가중치, 월간 실적, 제출 상태 |
| TEAM_MANAGER | 팀원이 제출한 KPI와 점검해야 할 월간 흐름 | 팀원 검토 queue, 반려/확정 필요 건, AI 코칭 |
| HQ_MANAGER | 본부 단위 진행률과 조정 필요 현황 | 본부 KPI alignment, 조직별 진행률, 등급/평가 preview |
| HR_ADMIN | 전체 오픈 readiness와 운영 blocker | MBO missing, confirmed KPI shortage, evaluator blockers, Team KPI pending |

상위 권한은 하위 권한 기능을 포함하되, 화면 첫 영역은 자신의 역할에서 가장 중요한 일로 맞춘다. 예를 들어 HR_ADMIN도 개인 KPI 메뉴를 볼 수 있지만, 첫 화면에서는 HR 운영 상태가 먼저 보여야 한다.

## 7. 평가 흐름 기반 IA

PPT의 평가 상태 흐름은 화면 전체의 공통 상태 모델로 사용한다.

| 상태 | 사용자 설명 | 주요 CTA | 수정 가능 범위 |
| --- | --- | --- | --- |
| 미시작 | 아직 작성이 시작되지 않음 | 작성 시작 | 작성 가능 |
| 작성중 | 초안 작성 중 | 저장, 제출 요청 | 작성자 수정 가능 |
| 제출완료 | 리더 검토 대기 | 제출 취소 또는 대기 | 제한적 수정 |
| 평가중 | 리더/평가자 검토 중 | 검토, 반려, 확정 | 평가자 권한 |
| 반려 | 보완 요청됨 | 수정 후 재제출 | 작성자 수정 가능 |
| HR 반영중 | HR이 운영/정책 반영 확인 중 | HR 검수 | HR 중심 |
| 1차확정 | 1차 평가가 확정됨 | 조정 검토 | 제한 |
| 조정중 | 본부/HR 조정 중 | 조정 review | HR/HQ 중심 |
| 최종확정 | 최종 확정됨 | 조회, 리포트 | 읽기 전용 |

### 상태 UX 규칙

- 모든 주요 row/card에는 상태칩을 붙인다.
- 상태칩은 색상만으로 의미를 전달하지 않고 텍스트를 함께 표시한다.
- disabled 버튼에는 `왜 비활성인지`를 짧게 보여준다.
- official write가 막힌 경우 `안전 잠금`, `preview`, `공식 저장 안 함`을 명시한다.

## 9. 영역별 개편 방향

### 조직 KPI

- 상단 큰 요약 박스를 줄이고 `조직`, `연도`, `상태`, `남은 작업`만 compact하게 둔다.
- 우측 긴 패널은 `요약`, `검토 이력`, `연결 개인 KPI`, `정책 안내` accordion으로 분리한다.
- `목표 일괄 수정`은 제거 방향으로 잡고, `목표 일괄 확정`은 HR/직책자 검토 흐름에 맞게 유지한다.
- 직책자는 조직 KPI를 본인 KPI처럼 자연스럽게 보고, 연결 가능한 개인 KPI 작성 진입점을 제공한다.

### 개인 KPI

- 영업/비영업 유형을 고려한 안내와 필드 강조를 분리한다.
- 핵심 입력은 `KPI명`, `정의`, `T/E/S 목표`, `가중치`, `정책 카테고리`로 정리한다.
- `연결 조직 KPI`, `태그`, `증빙 메모`는 선택/고급 영역으로 둔다.
- 기간이 닫혔을 때는 입력 form을 숨기기보다 `왜 닫혔는지`, `누구에게 문의할지`를 보여준다.

### 월간 실적

- 매월 상세 보고서를 쓰게 하는 느낌을 줄인다.
- 기본 입력은 `이번 달 진행 요약`, `핵심 증빙 링크`, `다음 달 위험/지원 요청` 정도로 단순화한다.
- AI는 구성원에게 `요약/표현 정리` 중심으로 보이고, 리더 리뷰 AI는 직책자에게만 보인다.

### 업적평가 / 평가 워크벤치

- `/evaluation/performance`는 HR 운영 dashboard와 평가 흐름 안내 역할을 유지한다.
- `/evaluation/workbench`는 preview-only를 명확히 유지하고, 공식 score/grade write와 분리한다.
- 팀원/팀장/본부장/HR별로 같은 평가 흐름을 다른 CTA로 보여준다.
- score/grade preview는 `공식 저장 안 함` 안내와 함께 표시한다.

### AI 활용 제출 / AI 지원 평가

- `AI 역량평가`라는 표현은 공식 평가 점수 산정처럼 보일 수 있어, `AI 활용 제출`, `AI 활용 실적`, `AI 지원 평가` 중 하나로 변경을 검토한다.
- 2028 Pass/Fail 요건과 2026 연간 평가점수 제외를 분명히 표시한다.
- AI 프로젝트 T/K, 조직 기여 AI 사례, AI 실무 인증 경로를 구분한다.

### 360 다면평가

- 평가 링크와 문항 나열보다 `응답해야 할 대상`, `남은 기간`, `익명/활용 방식`을 먼저 보여준다.
- 결과 화면은 HR/본부장용 요약과 개인/리더용 피드백을 구분한다.

### 리더십 진단

- `진단 참여`, `결과 확인`, `개선 액션`의 순서로 재배치한다.
- 리더에게는 본인 결과보다 다음 행동과 follow-up action이 먼저 보여야 한다.

### 관리자 설정

- `/admin/evaluation-ops`는 운영 허브로 유지하고, `오늘 해야 할 설정`, `위험한 고급 기능`, `참고 링크`를 분리한다.
- 평가 기간, 대상자, 평가자 매칭, 등급 기준, 조직/권한 관리를 한 흐름으로 묶는다.
- official transition, backfill, apply, feature flag 성격의 기능은 기본 화면에서 강조하지 않는다.

## 10. AI 기능 배치 원칙

| AI 유형 | 대상 | 위치 | 안전 문구 |
| --- | --- | --- | --- |
| 개인 KPI 작성 보조 | USER | `/kpi/personal` 작성 영역 | 개인 KPI 작성과 표현 정리를 돕는 보조 기능 |
| 월간 실적 요약 | USER | `/kpi/monthly` 월간 record 영역 | 저장 전 초안이며 사용자가 직접 확인 |
| 중간 점검 코치 | TEAM_MANAGER 이상 | 관리 범위 팀원 KPI/monthly 영역 | 공식 평가 점수나 등급을 산정하지 않음 |
| 리더 리뷰 초안 | TEAM_MANAGER 이상 | 리더 review context | 관리 범위 내 검토 보조 |
| AI 활용 제출 | USER/HR | `/evaluation/ai-competency` | 2026 연간 평가점수 제외, 2028 요건 준비 |

구성원 화면에는 리더/HR 전용 AI 기능을 숨긴다. disabled 상태로도 노출하지 않는다.

## 11. 공식 평가 안전 경계

이 설계는 official scoring/grade/write를 열지 않는다. UI 개편 중에도 다음 원칙은 유지한다.

- 공식 `Evaluation.totalScore` 저장 금지
- 공식 `Evaluation.gradeId` 저장 금지
- official `Evaluation` / `EvaluationItem` 생성 금지
- backfill/apply/dry-run 실행 버튼을 일반 운영 CTA처럼 노출 금지
- readiness와 workbench는 preview/read-only 메시지를 유지
